# Care Adaptive Experience

**Version 1.0**

> Care should never ask a first-day user to behave like a hundred-day user.

---

## 1. Philosophy

Care is a product about inner experience. It should feel like it knows you — not immediately, but gradually. A new user needs invitation. A long-term user needs space.

Most wellness apps make the same mistake: they show everyone the same interface. The heavy information architecture that helps a long-term user navigate efficiently overwhelms a new user who has no context for what any of it means. The sparse interface designed for a long-term user who "knows why they are here" leaves a new user disoriented.

The answer is not one UI that tries to serve everyone. The answer is an experience that listens.

**Care does not become calm by removing everything.**  
**Care becomes calm by learning what the user no longer needs.**

This means: home is not a static layout. Home is a page that changes as the relationship between Care and the user deepens.

---

## 2. User Phases

Care defines three phases of relationship:

### Phase 1 — Discovery

**Signal:** fewer than 3 check-ins

The user is new. They may not know what Care is. They may not have established any habit. They need:

- An invitation, not a dashboard
- Permission to move slowly
- A sense that something quiet and interesting exists here
- Orientation toward the ritual path, but without pressure

Home during Discovery is the most **inviting** — richer in explanatory content, quieter in its demands.

### Phase 2 — Relationship

**Signal:** 3–14 check-ins

The user has returned several times. They have drawn a card, journalled, checked in. A pattern is forming. They need:

- A sense that Care is building with them
- Today's context: what came to them, what they wrote
- Memory as a companion rather than a report
- A natural path toward the day's ritual

Home during Relationship is the most **present** — it knows what they've done and references it.

### Phase 3 — Companion

**Signal:** 15+ check-ins

The user has established relationship with Care. They return with intention. They do not need orientation, explanation, or prompting. They need:

- The date
- One invitation
- One action
- Possibly: one quiet memory

Home during Companion is the most **quiet** — it trusts the user completely.

---

## 3. What Each Phase Needs Emotionally

| Phase | Emotional need | What home provides |
|---|---|---|
| Discovery | "I am welcome here" | A warm, open invitation with no pressure |
| Relationship | "Care sees me" | Context from their own history, today's page |
| Companion | "I know why I am here" | Space, date, one action |

The shift is not from complexity to simplicity. It is from **invitation** to **presence** to **trust**.

---

## 4. What Home Shows in Each Phase

### Discovery Home

1. Date stamp (tiny, uppercase, quiet)
2. Large invitation card (paper surface, grain, lifted shadow)
   - Opening prose: "วันนี้... ลองกลับมาหาตัวเองสักครู่ไหม"
   - Body: explanation of Care's unhurried quality
   - CTA: เริ่มวันนี้ → /checkin
3. Ritual path (horizontal, text-only, with arrows)
   - เช็คอิน → การ์ด → บันทึก
   - Footnote: "ไม่จำเป็นต้องทำทุกอย่างในวันเดียว"
4. Collection teaser (very faint, below a soft divider)
   - "การ์ดบางใบจะค่อย ๆ เดินทางมาหาคุณ"
   - Quiet link to /collection

### Relationship Home

1. Date stamp
2. Header: greeting or missYou line
3. Card that came to them (paper surface, category, title, excerpt)
   - Link: "การ์ดที่เคยเดินทางมาหาคุณ"
4. Journal status (flat text, no card surface)
   - If written: excerpt + link
   - If not: "ยังมีพื้นที่ว่างสำหรับวันนี้"
5. Memory note OR Care Letter teaser (one only, left-bordered)
6. Primary CTA: เช็คอินวันนี้

### Companion Home

1. Date stamp
2. One sentence (greeting or missYou line)
3. Primary CTA: เริ่มวันนี้
4. Optional memory margin note (very faint, pencil-like)

## 5. What Home Removes Over Time

| Element | Discovery | Relationship | Companion |
|---|---|---|---|
| Large invitation card | ✓ | — | — |
| Care explanation prose | ✓ | — | — |
| Ritual path walkthrough | ✓ | — | — |
| Collection teaser | ✓ | — | — |
| MissYou greeting | — | ✓ | ✓ |
| Card preview (paper surface) | — | ✓ | — |
| Journal status | — | ✓ | — |
| Memory/diary note | — | ✓ | ✓ (fainter) |
| Quick actions grid | — | — | — |
| Stats/streak/totals | — | — | — |
| Secondary nav grid | — | — | — |

Stats, streaks, and grids are **never shown** on home in any phase.

---

## 6. Rules for Progressive Simplicity

1. **Phase signal is check-in count, not time.** A user who returns after a long pause and checks in once remains in Discovery. The phase advances through actual engagement, not calendar time.

2. **Phases do not regress.** Once a user enters Companion, they remain there even if they are away for weeks. The missYou state handles re-entry warmth independently.

3. **No phase label is ever shown to the user.** Phases are invisible. Users experience the result, not the mechanism.

4. **Rich does not mean noisy.** Discovery is the most content-rich phase, but it has one primary CTA, one invitation, and one path. It is rich in meaning, not in widgets.

5. **Quiet does not mean empty.** Companion has three elements. Each one is essential. Removing any would leave the user without orientation, without warmth, without action.

6. **Measurement language never appears on home in any phase.** No stats. No streaks as achievement numbers. No totals. No "12/40 collected." No completion language.

7. **Every surface on home uses paper tokens.** Paper tones, grain, borders, and shadows from the Care Paper Language System.

8. **Motion: care-page-enter wraps every phase.** The surface settles in; it does not pop, zoom, or spring.

---

## 7. How Care Avoids Overwhelming New Users

Discovery phase constraints:

- **One primary CTA** — เริ่มวันนี้ links only to /checkin. No secondary actions compete.
- **No stats at all** — a new user with 0 checkins does not need to see that 0.
- **No grid layouts** — no 2×2 or 2×3 quick-action grids that imply "many things to do."
- **Explanation embedded in prose** — Care's nature is explained through copy in the invitation card, not through tooltips or onboarding flows.
- **Collection shown as mystery** — "การ์ดบางใบจะค่อย ๆ เดินทางมาหาคุณ" creates curiosity without obligation.
- **Ritual path is descriptive, not prescriptive** — showing เช็คอิน → การ์ด → บันทึก as a path, not a checklist. No checkboxes. No "step 1 of 3."
- **No completion pressure** — "ไม่จำเป็นต้องทำทุกอย่างในวันเดียว" explicitly releases the user from urgency.

---

## 8. How Care Becomes Quieter for Returning Users

Relationship phase: the invitation card disappears. The ritual explanation disappears. In their place: the user's own content. Their card. Their words. The relationship is referenced, not explained.

Companion phase: even the card preview disappears. What the user wrote disappears. What remains is the date, a greeting, one button, and one faint note from memory. The user has internalized the ritual. They do not need to see their history to feel it.

The progression is:
- Discovery → "Here is what Care is and how to begin."
- Relationship → "Here is what you have done and what awaits."
- Companion → "You are here. Begin."

---

## 9. Risks

**Risk 1: Phase transition feels abrupt.**
The move from Relationship (14 check-ins) to Companion (15+) removes substantial content in one step. If the user notices, it might feel like a feature was removed.

*Mitigation: The missYou state provides warmth at phase transitions when the user has been away. The memory note in Companion preserves continuity with their history.*

**Risk 2: Phase 3 feels too empty for some users.**
A Companion user who wants more access to their history or collection has to navigate via bottom nav.

*Mitigation: The bottom nav provides consistent access to all features. The Companion home is quiet by design — not broken. Users who want more will navigate deeper.*

**Risk 3: Phase signal (check-in count) may not reflect actual engagement.**
A user could check in daily without drawing cards or journalling. They would advance to Companion quickly without the depth of engagement Companion assumes.

*Mitigation: The phase logic is an approximation. Future refinement could use a composite signal (checkins + journal entries + cards drawn). For V1, check-in count is the simplest stable signal.*

**Risk 4: New users who land in Phase 1 multiple times see the same invitation.**
The Discovery invitation is static copy. A user who checks in once, leaves, and returns three weeks later would see the same opening prose again.

*Mitigation: The missYou state activates after 3 days away, which subtly acknowledges their return even in Discovery phase. Long-term Discovery copy variation is a future enhancement.*

## 10. Future Implementation Notes

**Phase signal refinement**
Consider a composite signal: `(checkins × 1) + (journalEntries × 2) + (cardsDrawn × 1.5) > threshold`. This weights depth of engagement over frequency of check-in. Requires no new database tables — all counts already fetched.

**Discovery variant for returning Phase 1 users**
If `totalCheckins > 0 AND missYou.shouldShow`, adjust the Discovery invitation slightly:
- Instead of "ลองกลับมาหาตัวเองสักครู่ไหม" → use "ยินดีต้อนรับกลับมา"
- Keep the rest of the Discovery layout unchanged

**Relationship to Companion soft transition**
Rather than a hard threshold at 15, consider cross-fading the Relationship content over 15–20 check-ins: fewer panels visible each time, until by 20 only the Companion layout renders.

**Phase-aware collection teaser**
In Phase 2, show one specific card from the user's collection as a quiet reference. "การ์ดใบ G-03 เดินทางมาหาคุณเมื่อ 3 วันก่อน." This deepens the remembrance language.

**Seasonal or temporal awareness**
A Companion user on New Year's Day, or the first anniversary of their Care account, could see a special quiet note rather than the generic greeting. This requires date comparison logic, not AI.

**Phase export for other surfaces**
Future: expose the phase from a shared `getUserPhase()` utility so diary, collection, and growth pages can also adapt their density to the user's relationship depth.
